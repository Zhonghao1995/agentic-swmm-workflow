>
> **Audience**: any agent (human or AI) working on this codebase. Subagents should read this before grepping for unfamiliar terms.
>

## What this project is

`aiswmm` is an agentic CLI for EPA SWMM stormwater modelling workflows. A user runs `aiswmm interactive` and converses with an LLM-planned agent that drives SWMM simulations, audits the runs, and accumulates reusable knowledge across sessions.

The project optimises for **auditability and reproducibility**:

- Every action leaves a manifest.
- Every run produces byte-identical output given the same INP across macOS / Linux / Docker (validated — see [docs/byte-identical-reproducibility.md](docs/byte-identical-reproducibility.md)).
- Modeling memory accumulates lessons that future agent calls can recall via explicit promotion verbs.

## Domain vocabulary

### Session

A user's interactive REPL invocation. Begins with `aiswmm interactive` and ends with `/exit` or Ctrl-D. Each Session has:

- A `session_id` (timestamped, e.g. `225305`).
- A session directory at `runs/<date>/<session_id>_<slug>_<kind>/` (kind = `chat` or `run`).
- Zero or more `Run`s nested under it.
- An `agent_trace.jsonl` recording every event.
- A row in the SQLite `sessions` table at `runs/sessions.sqlite`.

A Session is the unit `recall_session_history` searches over.

### Run

A single SWMM simulation. Lives in `runs/<date>/<NNNNNN>_<slug>_run/`. Contains:

- `model.rpt` — SWMM text report.
- `model.out` — SWMM binary output.
- `manifest.json` — runner manifest (metrics + sha256 of inputs/outputs + return code).
- `09_audit/` — post-run audit artifacts (see *Audit* below).
- Optional: `07_plot/` for rendered figures.

A Run is produced by the `swmm-runner` skill calling `swmm5` directly. A Run is contained by a Session.

### Case

A named, registered watershed (e.g. `tecnopolo`, `tod-creek`). Cases live in `cases/<slug>/` and provide stable identifiers across Runs and Sessions. A Run may be linked to a Case via `case_id` for memory recall across modelling sessions.

Not every Run has a Case — ad-hoc runs are valid.

### Provider

The LLM backend driving the planner. Two implementations exist:

- `claude_sdk` (**default**; routes through the local `claude` CLI's OAuth so a Claude Pro/Max subscription is enough — no separate Anthropic API key needed, zero marginal per-token cost). On macOS the subscription credential lives in the login Keychain, not a JSON file; detection probes `security find-generic-password -s "Claude Code-credentials"` (exit code only, never reads the secret). A model is optional — the SDK follows the subscription default.
- `openai` (opt-in; requires `OPENAI_API_KEY`, billed per token; model defaults to `gpt-5.5`). Selected only via explicit `provider.default = openai` / `--provider openai`, never silently when a subscription is present.

Provider abstraction lives in `agentic_swmm/providers/`. Concrete providers implement the `ChatProvider` Protocol in `providers/base.py`. The planner never imports a concrete provider directly — it goes through `providers/factory.make_provider(name, ...)`. Adding a backend is a `factory.SUPPORTED_PROVIDERS` + `make_provider` change only.

Authenticate via `aiswmm login` (subscription) or `aiswmm login --openai` (opt-in key). See the `~/.aiswmm/config.toml` `provider.default` field and `aiswmm model --provider <name>` for switching at the user surface; `aiswmm login --status` / `aiswmm doctor` report which provider and credentials are active.

### Tool

A capability the agent can invoke. Each Tool has a `ToolSpec` (declared schema, validator, handler). Tools fall into two categories:

- **Direct handlers**: pure Python in `agentic_swmm/agent/tool_registry.py` or `agentic_swmm/agent/tool_handlers/*.py`.
- **MCP-routed**: server processes under `mcp/<server>/`, invoked via the MCP pool. The handler is generated by an MCP factory.

The registry (`AgentToolRegistry` in `tool_registry.py`) is the single source of truth for which tools the planner can see. `compute_intent_signals` and `_VALID_MODE_ENUM` also live here for now (see `tool_registry.py` split work in PR #128 + Phase 2).

### MCP (Model Context Protocol)

A protocol for running tool implementations as separate processes. aiswmm uses MCP servers for `swmm-builder`, `swmm-network`, etc. The MCP pool (`agentic_swmm/agent/mcp_pool.py`) manages connection lifecycle so subprocess startup cost is amortized across tool calls.

### Memory (four stores with different mutation rules)

aiswmm has four memory stores; understand the difference before writing into any of them.

| Store | Path | Mutation rule | Used for |
| --- | --- | --- | --- |
| Session history | `runs/sessions.sqlite` | **Automatic** on every interactive event | `recall_session_history` lookups across past Sessions |
| Modeling memory (parametric / lessons / citations / storms / benchmarks) | `memory/modeling-memory/*.{jsonl,yaml}` | **Manual** — only via explicit verbs: `aiswmm gap promote-to-case`, `aiswmm cite`, `aiswmm calibration accept` | Cross-Session domain knowledge |
| RAG memory | `memory/rag-memory/` | **Manual** — explicit add | Research-style retrieval (PRD-07) |
| Run-local memory | inside each `runs/<date>/<id>/` | **Automatic** per Run | Provenance and audit artifacts for one Run |

The four-store split is intentional, not legacy sprawl. See `agentic_swmm/memory/__init__.py` for the facade and PRD-03 (private archive) for the architectural rationale.

### Permission Profile

The executor's policy for "must this tool call prompt the user?":

- `QUICK` (default): auto-approve any Tool flagged `is_read_only=True` in its `ToolSpec`; prompt for everything else.
- `SAFE` (`--safe` flag): prompt for every Tool call regardless of flag.

Permission decisions are surfaced to the user as `[Y/n]` prompts in the digest output and recorded on the result dict under `permission = {prompted, approved}` (see `agentic_swmm/agent/executor.py`).

### Provenance

The immutable record of "what happened in a Run". Lives in `experiment_provenance.json` produced by the audit skill. Never mutated after writing — the file format carries a `schema_version` so older Runs remain readable when the schema evolves.

### Audit

The post-run pipeline that consolidates a Run's artifacts (`rpt`, `out`, `manifest`, `stdout`, `stderr`) into:

- `experiment_provenance.json` (machine-readable, immutable).
- `experiment_note.md` (human-readable digest).
- `model_diagnostics.json` (deterministic SWMM screening checks).
- `comparison.json` (if `--compare-to` given).

Lives in `skills/swmm-experiment-audit/`. Invoked by `aiswmm audit --run-dir <path>`. **The audit pipeline never re-runs SWMM** — it only reads a Run's existing manifests.

### Workflow Mode

Historically a coarse classification (`prepared_inp_cli`, `gis_to_inp`, `calibration`, …) that the planner used to route to a skill family via a `select_workflow_mode` first-hop tool. **Removed in the LLM-driven dispatch refactor** (see *Dispatch architecture* below) — the LLM now reads each tool's description / SKILL.md and picks tools directly without a mode gate. The string still exists as an optional *tag* on `audit_run` payloads for provenance bookkeeping, but it is no longer a routing surface.

### Skill

A self-contained unit of capability that the agent can dispatch to. Each Skill lives in `skills/<name>/` with:

- A `SKILL.md` describing what it does and how to invoke it (the agent reads this file at runtime).
- Scripts in `scripts/`.
- Tests in `tests/`.

Examples: `swmm-runner` (executes SWMM), `swmm-experiment-audit` (post-run audit), `swmm-plot` (renders figures), `swmm-modeling-memory` (memory summarization).

### Intent

A classified user goal extracted from a prompt. Categories: `wants_demo`, `wants_run`, `wants_plot`, `wants_calibration`, etc. Defined in `agentic_swmm/agent/intent_classifier.py`. Used by:

- The warm-intro classifier (`is_open_shaped_prompt`) to decide whether a prompt looks open-shaped enough to need the warm-intro template.
- The memory-informed-policy hook before the LLM main loop, as one input to the case-resolution heuristic.

**Not** used for tool routing post-refactor — the LLM picks tools directly from their descriptions.

## Key seams

### CLI → Library

- `agentic_swmm/cli.py` is the entrypoint. It dispatches to verb modules in `agentic_swmm/commands/`.
- Verb modules MUST stay thin: argparse + help-text + a 1-2 line call into business logic.
- Business logic lives under `agentic_swmm/agent/`, `agentic_swmm/audit/`, `agentic_swmm/providers/`, `agentic_swmm/memory/`.

### Planner ↔ Provider

- The planner imports `agentic_swmm/agent/planner.py:OpenAIPlanner`.
- The planner constructs its Provider via `agentic_swmm/providers/factory.make_provider(name, ...)`.
- The planner only sees the `ChatProvider` Protocol (`providers/base.py`); it does not import concrete provider modules.

### Planner → Executor (via ToolCall)

- The planner emits `ToolCall` dataclasses to the executor.
- The executor (`agentic_swmm/agent/executor.py`) inspects each `ToolCall`, applies the permission profile, dispatches via the registry, and returns a result dict carrying:
  - The tool's output (raw).
  - Executor metadata under `permission = {prompted: bool, approved: bool}`.
- A denied tool call returns a sentinel summary and the executor's `permission.approved == False`. The digest renderer uses this to print `(skipped)`.

### Executor → Tool Registry → Handler

- The registry resolves a tool name to a handler function.
- Handlers are either:
  - **Direct**: a Python function in `tool_registry.py` or `tool_handlers/<family>.py`.
  - **MCP-routed**: a thin function generated by an MCP factory that proxies to a subprocess via `mcp_pool`.
- Tool handler functions return a dict; the digest renderer extracts a one-line `brief` via `digest_render.brief_result()`.

### Runtime → Memory layer

- `agentic_swmm/memory/session_db.py` writes Session events to SQLite automatically.
- Modeling memory (`memory/modeling-memory/`) is read freely by the runtime but only mutated via explicit promotion verbs.
- RAG memory is read freely and indexed manually.

### Session → Audit

- Audit is **post-run, never inline** with SWMM execution.
- `aiswmm audit --run-dir <path>` reads the Run's existing artifacts and produces audit outputs.
- The audit pipeline reads the runner manifest **exactly once** via `collect_run` (PR #194 cleanup). Do not re-introduce double-reads.

## Key invariants

1. **No tool runs without a permission check.** The profile auto-approves read-only tools; everything else prompts the user. This is the user's primary safety surface.
2. **Provenance is append-only.** `experiment_provenance.json` is written once per audit pass; never mutated. Schema changes require a new `schema_version`.
3. **The runner manifest is the source of truth for a Run's metrics.** Anyone needing peak / continuity reads `manifest.json` — do not re-parse the `.rpt`.
4. **Modeling memory only mutates via explicit verbs.** No agent action writes to `memory/modeling-memory/` autonomously. (Auto-distillation is an open research direction — see private PRD-12.)
5. **No `Co-Authored-By: Claude` trailer in any commit.** Forward-only — do not rewrite history. See global `~/.claude/CLAUDE.md`.
6. **Reproducibility is byte-level for SWMM execution.** The same INP must produce the same `model.out` byte-for-byte across macOS / Linux / Docker. Validated via `aiswmm tecnopolo` benchmark + the `tecnopolo` Docker entrypoint. Do not introduce nondeterminism in the runner skill.

## Real-data path vs synth-data path

A new orthogonal axis introduced by the `swmm-anywhere` skill (PRD `swmmanywhere_integration`). Two parallel entry paths for "I want a SWMM model":

| Aspect | Real-data path (`swmm-gis` / `swmm-network`) | Synth-data path (`swmm-anywhere`) |
| --- | --- | --- |
| User inputs | `.shp` / CAD / `network.json` of real pipes, or a prepared `.inp` | Just a bbox (4 numbers) — no pipe inventory needed |
| What aiswmm does | Imports the user's measured infrastructure into a SWMM model | Downloads OSM streets + DEM + buildings; SWMManywhere infers a plausible network |
| Reproducibility guarantee | byte-identical given same INP | byte-identical given same `00_raw/` snapshot (OSM is mutable upstream; snapshot pins it) |
| Best for | Catchments where the city has pipe data on hand | Data-scarce regions; pre-survey scoping; sensitivity baselines |
| Downstream skills | identical — both paths feed `swmm-runner` → `swmm-experiment-audit` → `swmm-plot` | identical (validated by the e2e chain in `scripts/spike_swmmanywhere/05_e2e_chain.py`) |
| What planners must NOT do | Use `swmm-anywhere` when the user has real pipe data — the synth network would silently disagree with the real one | Skip the `00_raw/` snapshot — OSM drift would invalidate reproducibility |

The orchestrator (`swmm-end-to-end`) picks the entry based on user inputs: any pipe file → real-data; bbox-only → synth-data. Output structure under `runs/<date>/<id>/` is intentionally identical from `swmm_run/` downward so audit / plot / calibration treat both paths symmetrically.

## Dispatch architecture: LLM-driven over hardcoded mode enum

### Decision

The planner sends the full `AgentToolRegistry.schemas()` to the LLM and lets the LLM pick tools by name. There is no `select_workflow_mode` first-hop gate, no `workflow_modes/<mode>.py` adapter registry, and no `intent_disambiguator` that re-derives a mode string from keywords. Tools live in `agentic_swmm/agent/tool_handlers/<family>.py` with typed-parameter validation, exactly the shape the OpenAI / Anthropic function-calling APIs assume.

### Context (why this was decided)

v0.7.0a2 introduced a defensive dispatch layer between the user prompt and the SWMM tools: a `select_workflow_mode` tool with a seven-value enum, per-mode adapter files, and an LLM-classifier helper that triggered on plot-conflict goals. The layer was a GPT-4-era guardrail that:

- Hid the concrete SWMM tools from the LLM behind one big "pick a mode" tool — the opposite shape from how every other tool-calling API works.
- Re-imposed keyword-matching brittleness (the legacy classifier was the fallback inside the mode tool itself).
- Cost ~4 files per new skill (an adapter, a handler, an enum value, and a disambiguator prompt update). The new `swmm-anywhere` synth-from-bbox path exposed the unsustainability: there was no `synth-from-bbox` enum value, so a "use SWMManywhere on this bbox" prompt always fell through to the wrong mode.

Frontier 2026-era LLMs (GPT-5.5 / Claude Opus 4.7+) pick the right tool from a flat registry at ~95% accuracy when each tool's description is well-written. The mode gate was throwing that capability away.

### Consequences

- **Adding a skill is one file.** Drop a typed handler under `tool_handlers/<name>.py`, register a `ToolSpec` in `tool_registry._build_tools()`, write a SKILL.md for the LLM to read. No adapter / enum / disambiguator edits.
- **The LLM's tool-pick decision is auditable as a single event** — `planner_response.tool_calls` in `agent_trace.jsonl` — instead of a tool-call chain (`select_workflow_mode` → adapter → real tool) that hid which classifier made the decision.
- **Two-level select_skill flow stays.** The `select_skill` tool that exposes deterministic-SWMM skills is unchanged; the LLM picks the skill, then picks the tool within it. That's a *help-the-LLM* surface, not a routing gate.
- **Memory-informed-policy hook stays.** The pre-LLM consult (`_consult_memory_informed_policy`) that escalates high-stakes goals to HITL is orthogonal to dispatch and survives unchanged.

The decision record lives at `.claude/prds/PRD_llm_driven_dispatch.md`.

## Private design intent

Research-sensitive design directions (e.g. auto-distillation of chat sessions into modeling memory, novel memory architectures backing the academic paper) live in a separate private archive at `~/Desktop/aiswmm-private-prds/` on the maintainer's machine and the corresponding private GitHub repo. **Do not surface those decisions in this public CONTEXT.md** until they're ready for public release — that risks academic novelty erosion.

If you (a future subagent or contributor) need to understand a decision and can't find it in `CONTEXT.md` or `docs/adr/`, surface that gap as a finding rather than guessing.
